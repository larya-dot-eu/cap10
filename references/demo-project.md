# Demo Project

Use this when a user wants to see Cap10 in action before committing their
own project, or on a clear first-run signal. This is a compressed, scripted
walkthrough, not a full game.

## What it demonstrates
Project: "Build a to-do list app that judges you." Chosen because it needs
no domain explanation, is small enough to honestly scope in a handful of
stages, and the joke is built into the premise rather than bolted on.

## Output format note
Markdown (the tracker file and native task list) is the default output for
every game — fast, zero-dependency, readable anywhere. If a user wants a
real visualization instead of plain text (a rendered tree or progress
diagram), that's available on request as a generated file, not a default.
Mention this once during the demo so new users discover the option without
it becoming the default experience for everyone else.

## Compressed intake (3-4 questions, not 10)
Ask a short, representative subset — enough to show the question-count
mechanic firing without making a demo user sit through ten questions for a
joke project. Label it clearly as a shortened preview of the real intake.

## Compressed hierarchy (not a full 10x10 walkthrough)
Use a small real hierarchy — around 3 stages, 2-3 phases each, no deeper
drilling needed. This should demonstrate the propose -> lock -> tracker
update loop honestly, including showing a stage that deliberately uses
fewer than 10 phases, to model the "cap is a ceiling, not a target" rule
directly rather than just stating it.

## Completion moment
Run the real completion display: final tree, unit counts versus the
theoretical maximum, Trade Log if applicable, questions used out of 10, then
one dry closing line per references/voice.md. This is the part of the demo
that actually has to land — it is the entire pitch for why someone would
play a full game afterward.
